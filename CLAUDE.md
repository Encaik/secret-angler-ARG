# CLAUDE.md — 《秘境垂钓者》项目规则

## 项目概述

《秘境垂钓者》是一款以网站为载体的浏览器端 ARG 解谜游戏。游戏主体是一个看起来像真实爱好者社区的网站——"深空古迹探秘导航站"。项目使用 Astro 静态站点生成器，输出纯静态 HTML/CSS/JS。

## 技术栈

- **框架**: Astro 4.x（静态站点生成器，MPA 多页应用）
- **语言**: TypeScript（strict mode）
- **样式**: 纯 CSS（手写，复古小众站点风格，禁用 Tailwind）
- **客户端逻辑**: 原生 JavaScript / TypeScript
- **状态**: localStorage
- **部署**: Vercel / Netlify 静态托管

## 代码架构规则

### 目录职责

```
src/
├── pages/        # Astro 页面，每个 .astro 对应一个路由
├── components/   # 可复用 Astro 组件
├── scripts/      # 客户端 TypeScript（谜题、进度、结局）
├── data/         # 游戏内容数据（纯 TS 对象，不含 UI 逻辑）
├── styles/       # CSS 样式文件
└── assets/       # 静态资源（图片、下载文件）
public/           # 直接映射到根路径（robots.txt 等）
```

### 组件层级

- `Layout.astro` — 全局布局（导航栏、页脚），所有公开页面使用
- `HiddenLayout.astro` — 隐藏/后台页面专用布局（暗色主题）
- 页面组件放在 `pages/` 目录，按板块分目录
- 共享 UI 组件放在 `components/` 目录

### 页面布局规范

- **所有页面必须使用统一的页面框架。** 公开页面使用 `Layout.astro`，隐藏页面使用 `HiddenLayout.astro`，禁止页面自行编写全局布局结构（导航、页脚）
- **页面框架宽度统一。** 导航栏、页脚、主内容区均通过全局 CSS 变量 `--max-width`（当前值 `1020px`）控制。**宽度容器必须同时设置 `width: 100%` 和 `max-width`**，仅设置 `max-width` 而不设 `width` 会导致内容少的页面容器坍缩到内容宽度（如搜索页实测仅 306px）
- **内容区宽度分两类：**
  - **内容型页面**（帖文、列表、搜索、用户主页等）：内容应填满框架宽度，禁止额外设置 `max-width` 限制（如 `max-width: 700px` 等）
  - **表单/认证型页面**（登录、注册、投稿、权限提示等）：允许在框架内使用较窄的居中容器（建议 `400-500px`），模拟真实网站的登录卡片效果
- **页面框架最小宽度不小于 `960px`。** 低于此断点时应触发横向滚动条或响应式收窄，而非让布局破碎
- 内容区内边距统一使用全局规范（当前 `padding: 20px 16px`），页面级样式不得覆盖
- 如需两栏/三栏布局，使用全局定义的 `.two-col` 等栅格类，避免页面内临时定义宽度
- **全局 CSS 中 `input, textarea, select { max-width: 400px }` 会限制所有表单元素。** 搜索框等需要全宽的场景必须显式覆盖 `max-width: 100%`，否则会被压缩到 400px

### 状态管理

- 所有游戏进度存储在 localStorage，键名 `arg_progress`
- 谜题答案仅存储 SHA-256 哈希值
- 状态读写通过 `src/scripts/progress.ts` 统一接口
- 跨标签页状态通过 `storage` 事件同步

### 路径引用规范（兼容根目录与二级目录部署）

项目同时支持根目录部署（本地开发 `pnpm dev`）和 GitHub Pages 二级目录部署（`BASE_URL=secret-angler-ARG pnpm build`）。

**核心机制：**
- `astro.config.mjs` 通过 `BASE_URL` 环境变量拼接 `base` 路径（`/repo-name/`），同时通过 Vite `define` 注入全局常量 `__BASE_PATH__`
- `.astro` 文件 frontmatter 中定义 `const BASE = __BASE_PATH__;`，模板内所有 `href` 使用模板表达式：`href={`${BASE}home/`}`
- 客户端 `.ts` 脚本中同样使用 `const BASE = __BASE_PATH__;`（Vite 构建时替换为实际字符串）
- 内联 `<script>` 块中使用 `const BASE = __BASE_PATH__;`（不带引号/大括号包裹——它是 Vite define 标识符，不是 Astro 模板表达式）

**强制规则：**
- **禁止在 `href`、`src`、`action` 等属性中硬编码绝对路径**（如 `href="/home/"`、`src="/src/scripts/xxx.ts"`）
- **禁止使用 `import.meta.env.BASE_URL`**——Windows Git Bash 下 MSYS2 会将其转译为文件系统路径（如 `C:/Program Files/Git/...`）。统一使用 `__BASE_PATH__`
- **`window.open()` 和 JS 内字符串拼接路径也必须加 BASE 前缀**：`window.open(BASE + 'member/', '_blank')`
- 数据文件（`src/data/*.ts`）中存储的 `pageUrl`、`authorUrl` 等路径字段保持根相对格式（`/user/xxx/`），在模板/脚本渲染时动态拼接 BASE
- 内联脚本中 `${BASE}xxx` 的 JS 模板字面量用法必须是 JS 变量 `BASE`（小写），而非 Astro 模板表达式

**构建命令：**
```bash
# 本地开发（根目录）
pnpm dev

# GitHub Pages 部署（二级目录）
BASE_URL=secret-angler-ARG pnpm build
```

**TypeScript 声明：** `src/env.d.ts` 中已声明 `declare const __BASE_PATH__: string;`

## 命名规范

### 文件命名

- Astro 页面/组件：kebab-case（如 `search-bar.astro`）
- TypeScript 脚本：kebab-case（如 `puzzle-verify.ts`）
- 数据文件：kebab-case（如 `shenci-journal.ts`）
- CSS 文件：kebab-case（如 `home.css`）

### 变量命名

- 谜题 ID：snake_case（如 `stage1_login_password`）
- 页面路由：kebab-case（如 `/alien/sighting-001`）
- 函数命名：camelCase（如 `verifyAnswer`）
- localStorage 键名：snake_case（如 `arg_progress`）

## MPA 设计规则

- **所有内部链接使用 `target="_blank"`**，在新标签页打开
- 每个页面是独立的 HTML 文件
- 导航体验模拟真实网站浏览
- 不创建 SPA 路由，不使用客户端导航

### 静态优先原则（沉浸感高于路由优雅）

**本项目是纯静态站点，且必须永远保持静态。** 游戏中的"动态"行为（登录态切换、权限判断、内容变化等）全部通过以下方式模拟，绝不引入服务端逻辑：

| 需求 | 静态模拟方案 |
|------|------------|
| 登录态切换内容 | localStorage + JS 切换 `display` / 替换文本 |
| 不同用户看到不同内容 | 同一页面内嵌入多份 HTML 区块，JS 按 localStorage 状态显示对应区块 |
| 权限门控 | JS 读取 `arg_progress` 进度标记，未解锁时显示"权限不足" |
| 搜索触发隐藏页面 | 客户端关键词匹配 → `window.open()` |
| 表单提交后的不同结果 | 同一表单，JS 判断输入内容 → 显示不同回复（无需服务端） |

**如果某个功能在动态站点中只需一个路由，但在静态站点中需要拆成多个页面——拆。** URL 多几个无所谓，沉浸感不破是底线。

**反例（禁止）：**
- ❌ 引入 SSR/Edge Function/API Route 来实现"真正的登录"
- ❌ 为了 URL 干净而把所有状态逻辑塞进一个页面，导致 JS 分支爆炸、代码不可维护
- ❌ 用 query string（`?loggedin=true`）切换内容——真实网站不会这样暴露状态
- ❌ 因为"两个状态只差一行文字"就复用同一个页面——细微差异也应拆成独立页面，模拟真实网站的多版本缓存

**正例（推荐）：**
- ✅ 沈辞隐藏帖文：非本人看到"权限不足"，沈辞本人看到正文——同一页面，两个 HTML 区块，JS 切换
- ✅ 评论区登录态切换：登录前后看到不同评论——同一页面，`display:none/block` 切换
- ✅ 暗网交易市场：需要账号密码才能进入——独立页面 `/hidden/darknet/`，JS 验证 localStorage 中的暗网账号
- ✅ 会员专区：需要解锁码——独立页面 `/member/` + `/member/login/`，JS 验证进度标记

**URL 命名原则：** URL 模拟真实网站的路径结构即可，不需要刻意追求"RESTful"或简短。多几个路径片段（如 `/community/post/247/`）比 `/post-247` 更像真实网站。

### 多用户页面拆分原则

**每个用户的专属视角页面必须拆分为独立静态文件，禁止用 JS 在单一页面内动态切换用户内容。** 真实网站中，每个用户的个人中心、主页、设置页都是独立的 URL——`/user/center/lyu/` 和 `/user/center/shenci/` 是两个不同的页面，而非同一个页面根据 localStorage 显示不同内容。

**规则：**
- ✅ 有多少个用户，就创建多少个静态页面——哪怕目前只有 lyu 和 shenci 两个
- ✅ 登录后通过 `window.location.replace()` 重定向到对应用户的静态页，不在同一 URL 内 JS 切换
- ✅ 每个用户页面的内容（用户名、头像色、帖文列表、注册日期）全部硬编码为静态 HTML
- ❌ 禁止把所有用户数据塞进一个 JS 对象，在客户端根据 `arg_username` 动态渲染
- ❌ 禁止用 `?user=lyu` 或 `#lyu` 等 query/hash 参数区分用户

**已拆分的页面（当前实现）：**

| 页面 | 路由 | 用户 |
|------|------|------|
| 林屿个人中心 | `/user/center/lyu/` | lyu |
| 沈辞个人中心 | `/user/center/shenci/` | shenci |
| 个人中心入口（未登录引导） | `/user/center/` | 通用 |

```
src/pages/user/center/
├── index.astro   → /user/center/      未登录引导（检测已登录则自动跳转）
├── lyu.astro     → /user/center/lyu/  林屿个人中心（纯静态）
└── shenci.astro  → /user/center/shenci/ 沈辞个人中心（纯静态）
```

**此原则同样适用于未来可能新增的任何"用户相关"页面**，包括但不限于：用户设置页、收藏页、草稿箱等。

## 内容创作规则

### 语言

- 所有面向玩家的文本使用**简体中文**
- 代码注释优先使用中文（方便策划阅读理解）
- Git 提交信息使用中文

### 叙事一致性

- "林屿"视角：谨慎细腻的观察者（太空/外星爱好者，理论派）
- "沈辞"视角：大胆执着的行动派（古迹/历史爱好者，实地派）
- 所有看似"超自然现象"必须在深层页面揭示为人工骗局
- 线索碎片化：每个页面提供部分信息，拼合才能理解全貌

### 谜题线索规范

- 每个谜题至少提供 2 条不同维度的线索
- 线索来源多样化：页面正文、图片隐写、登录态切换揭示、搜索框交互、URL 路径推断等
- 引导方式：暗示而不明示（如"留白处藏通路"而非"查看网页源代码"）
- 谜题难度逐阶段递进

### 谜题设计原则——密室逃脱风格

**核心理念：所有谜题应模拟"密室逃脱/实体解谜游戏"的体验，而非 CTF 竞赛或编程挑战。目标玩家是普通解谜爱好者，不是程序员或密码学爱好者。**

- **禁止使用专业技术编码。** 谜题中不应出现 Base64、凯撒密码、摩斯密码、哈希算法、XOR 运算等需要计算机/密码学专业知识才能理解的编码方式。如需使用编码，必须在页面内提供完整的解码工具或极明确的解码步骤（但优先避免使用）
- **优先使用通用的文字/视觉谜题。** 推荐类型包括：
  - 藏头诗/藏头文（中文传统文字游戏，人人能理解）
  - 文字排序与提取（首字连读、特定位置取字等）
  - 符号/颜色对应（页面中的视觉元素映射）
  - 日期/数字规律（日历推理、时间线拼接）
  - 对话中的首字母/首字提取
  - 图片中的视觉线索（图案匹配、拼图、迷宫等）
- **谜题应基于观察力而非知识储备。** 玩家需要的应该是"仔细看"而不是"学过这个"。线索藏在内容的排列方式中，而不是藏在专业术语里
- **提供多层提示。** 每个谜题至少有三个独立维度的提示，确保玩家不会因单一视角卡关。提示应嵌入叙事/角色对话中，而非直接给出解码指令
- **文化亲近感优先。** 优先选择中文文化中熟悉的谜题形式（藏头诗、字谜、对联等），而非西方密码学传统形式
- **禁止隐写术形式的 "专业工具链"。** 不使用 LSB 图片隐写、频谱分析、文件头分析等需要专用软件的技术。如果使用图片隐藏信息，应通过视觉对比、裁剪拼接等肉眼可辨识的方式
- **谜题难度表现为"线索分散程度"而非"技术复杂度"。** 简单谜题 = 线索集中在一个页面；困难谜题 = 线索分散在多个页面/多次搜索中，但每个单独的线索都是可理解的

**检查清单**（新谜题上线前必须逐项确认）：
- [ ] 一个完全不懂编程的普通玩家，能在合理时间内解出这个谜题吗？
- [ ] 谜题的核心乐趣是"发现"和"顿悟"，而非"计算"或"解码"吗？
- [ ] 至少有 3 条独立线索指向同一个答案吗？
- [ ] 如果玩家卡住了，他们能通过搜索其他关键词找到更多提示吗？

### 页面内闭环规则

- **游戏全程不得脱离页面本身。** 所有谜题的发现、推理、解答必须在浏览器正常浏览范围内完成
- **禁止依赖浏览器开发者工具（F12）。** 线索不应藏在 HTML 源码注释、CSS 样式、Console 控制台消息等需要开发者工具才能看到的位置
- **禁止依赖外部工具。** 玩家不需要使用 Python、在线解码网站、Photoshop 等页面外的工具。如需编码/解码，应在页面内提供隐式的解码方式（如搜索框、页面内的文本暗示等）
- **iframe / 嵌入内容除外。** 可以接受图片、PDF 等浏览器原生支持的嵌入内容，但这些内容中的线索必须同样遵守以上规则
- 已违反此规则的内容（如 HTML 注释中的 Base64）需要迁移到页面可见区域或通过交互触发显示

### 搜索功能核心定位

- **搜索是整个游戏的核心串联机制。** 玩家通过搜索发现隐藏内容、历史帖文、线索碎片
- **页面可见 = 近期，搜索可见 = 历史。** 各页面直接展示的内容按设定为近 3-6 个月的帖文/动态；更早的历史内容（2025 年及以前）仅能通过搜索关键词检索到
- 搜索关键词可以匹配出页面链接中不直接出现的历史帖文摘要、早期讨论、已沉底的内容
- 关键线索（如角色早期经历、密码的组成部分）应分散在可通过搜索发现的历史内容中，而非直接写在近期页面上

### 搜索行为铁律（沉浸感底线）

**搜索绝不直接跳转。** 真实网站的搜索框输入关键词后，一定是先出现搜索结果页，用户在结果中看到链接后自行点击。关键词→直接跳转到隐藏页面，这是游戏作弊码的逻辑，不是网站的逻辑。

- ❌ 禁止：搜索"深海裂隙"→ 页面直接跳转到 `/trigger/rift`
- ✅ 正确：搜索"深海裂隙"→ 搜索结果页显示一条"缓存页面"记录，标题是"[搜索引擎缓存] 内部系统操作日志"，用户点击标题链接 → 访问 `/trigger/rift`
- ❌ 禁止：搜索"轻舟云盘"→ 直接打开云盘页
- ✅ 正确：搜索"轻舟云盘"→ 搜索结果页显示两条结果：①星海漫步的推荐帖文 ②一个"外部服务"缓存条目链接到 `/share/`

**唯一的例外**：谜题答案（如"深海裂隙2026"）解锁进度标记后可以跳转到对应页面——这模拟的是"输入正确密码后进入"的体验，而非搜索行为。

**实现方式**：
- 隐藏页面的入口信息全部放在 `search-data.ts` 的 `HIDDEN_PAGE_CACHE` 中，作为"缓存页面"类别的搜索结果
- 搜索时匹配到关键词 → 在搜索结果页中显示 `pageUrl` 为可点击链接的标题
- `search.ts`（导航栏搜索框）和 `search.astro`（搜索结果页）的搜索逻辑一致：都走搜索结果页，不做直接跳转

### 加粗规范

- 页面中与解谜相关的**关键信息**必须加粗（`<strong>`），帮助玩家在大量文字中定位重点
- 为避免"加粗=答案"太明显，每个页面应**额外加粗 1-2 处有用但与解谜不直接相关的内容**作为混淆
- 示例：一篇博文中，真正指向密码的线索加粗了，同时地名、时间、装备名也可能加粗，玩家需要自行判断哪些是关键
- 加粗密度：每 200 字正文约 2-4 处加粗，避免过度使用导致无意义

### 动态内容规则

- **禁止静态写死登录态相关的 UI 文本**。如"仅登录用户可查看全部评论"这类文字，必须配合 JS 根据实际登录状态动态切换显示/隐藏
- 需要登录才能查看的内容，未登录时显示提示，登录后必须展示实际内容
- 评论数、在线人数等动态数据可以写死模拟值，但权限相关的 UI 逻辑不能造假
- 此规则同样适用于：私信入口、会员板块入口、个人中心入口等所有权限相关 UI

### 时间线约定

- 游戏"当前时间"锚定在 **2026 年**（即今年）
- 游戏中所有帖文日期、用户注册日期、站点建立日期均以此为准
- 站点成立：2021 年（距今约 5 年）

### 真实感原则

- 网站外观和用语参考真实国内小众论坛（牧夫天文论坛、8264户外等）
- 不使用现代设计框架（无 Tailwind、无 Material Design）
- 不用过于精致的动画和过渡效果
- 允许适当的信息密度和"粗糙感"

### 页面沉浸感规则（重要——所有页面必须遵守）

**游戏中的所有页面都在模拟真实网站。每个页面都是一个"真实的"网页——而不是一个"给玩家玩的谜题关卡"。这一原则高于一切。**

**禁止事项：**

- ❌ **禁止在页面中放置开发者解释型内容。** 包括但不限于："请保存证据并立即报警"、"你正在浏览的是一个犯罪平台"、"以下为关键发现"、"搜索xxx可找到更多信息"、"请将以下线索与xxx交叉验证"——真实网站上永远不会出现这些文字
- ❌ **禁止"💡 关键发现"式的总结框。** 不要替玩家总结。把信息碎片放在那里，让玩家自己拼
- ❌ **禁止在页面中用 HTML 注释向玩家喊话。** 注释中的"线索"必须迁移到页面可见区域
- ❌ **禁止在正文中使用游戏设计术语。** 如"谜题"、"线索"、"解谜"、"触发"、"跳转"等——真实网站的编辑不会写这些词
- ❌ **禁止在隐藏页面中写"引导下一步"的文字。** 如"搜索xxx了解更多"、"去xxx页面查看"——让信息本身的空缺引导玩家自行搜索

**应该做的事：**

- ✅ 每个页面读起来应该像它声称的东西——论坛帖文就像论坛帖文，公告就像公告，内部文档就像内部文档
- ✅ 暗网/后台页面应该模拟真实犯罪平台的语气——冰冷、功利、物化，没有任何道德评论
- ✅ 线索通过信息本身的"空缺"和"异常"来呈现——不是通过"这是线索"的标注
- ✅ 触发页面的"受限结果"、"缓存于"、"字段丢失"等伪装应尽可能逼真

**自查问题（每个页面发布前必须回答）：**
1. 这个页面上的文字，在它声称的那个真实场景中会存在吗？
2. 有没有任何一句话在"给玩家解释游戏机制"？
3. 如果我不告诉读者这是一个游戏，他们会不会觉得这个页面是真实的？

## 文档同步规则

**三份核心文档必须同步更新——游戏内容（代码/页面）、剧情逻辑（`game-design/剧情推进逻辑.md`）、主线流程（`game-design/主线流程.md`）是同一游戏的三份视图，任何一方的修改必须同步反映到另外两方。**

| 文档 | 职责 |
|------|------|
| 游戏代码/页面 | 玩家实际接触到的内容 |
| `game-design/剧情推进逻辑.md` | 叙事逻辑是否自洽（角色动机、时间线、信息层级） |
| `game-design/主线流程.md` | 流程是否可走通（谜题步骤、触发条件、阻塞点） |

- **游戏内容变更时必须同步检查并更新另外两份文档。** 包括但不限于：谜题线索位置变更、密码/答案变更、页面增删、流程逻辑调整、新增或修改搜索关键词、帖文日期/内容修改、角色设定变更
- 修改密码/答案 → 同步更新游戏代码（哈希值） + 主线流程（答案记录） + 剧情逻辑（线索描述）
- 修改时间线 → 同步更新游戏代码（帖文日期） + 主线流程（触发窗口） + 剧情逻辑（时间线全局一致性）
- 修改叙事逻辑（如发现链条、角色动机） → 同步更新游戏代码（页面内容体现） + 主线流程（流程步骤调整） + 剧情逻辑（逻辑描述）
- 流程文档是开发者追踪主线是否可走通的唯一参考，文档与代码不一致会导致误判阻塞点
- 更新流程文档时同步更新"最后更新"日期和"阻塞点"清单
- 新增谜题或线索时，在三份文档中分别写清：代码实现、线索位置/触发条件/验证点、叙事逻辑关联
