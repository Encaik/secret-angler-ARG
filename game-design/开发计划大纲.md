# 《秘境垂钓者》开发计划大纲（修订版）

---

## 一、项目概述与技术决策

### 1.1 项目定位

《秘境垂钓者》是一款**以网站为载体的浏览器端 ARG 解谜游戏**。游戏主体是一个看起来像真实爱好者社区的网站——**"深空古迹探秘导航站"**。玩家如同普通访客般浏览该网站，通过搜索、查看源码、解密隐藏内容、追踪线索，逐步揭开网站背后的灰产骗局真相。所有游戏体验均在浏览器中完成，无需模拟桌面。

### 1.2 核心设计思路

- **网站即游戏**：游戏就是一个真实的网站，玩家在其中的每一次点击、搜索、源码查看都是游戏行为
- **分层隐藏**：公开页面 → 源码隐写 → 关键词触发 → 隐藏后台，难度递进
- **无框架痕迹**：网站外观和代码风格贴近真实的国内小众爱好者社区，增强 ARG 沉浸感
- **叙事融入内容**：沈辞的笔记、日志、加密文件等，以"用户投稿"、"资源分享"、"个人主页"等形式呈现在网站上
- **广度伪装 + 深度聚焦**：网站包含 7+ 个探秘板块（外星、古迹、未解之谜、都市传说、自然奇观、深海探秘、地心文明等），但只有"外星目击"和"古迹秘境"两个板块有深层谜题线索。其余板块仅有 1-3 页表面内容。这种设计让网站看起来像一个真实的全品类探秘社区，而非专门为两个主题定制的"游戏道具"。玩家在广泛浏览中逐渐发现"只有这两个板块有深度"这一事实，本身就是一条隐藏的叙事线索——团伙只对他们目标人群的板块投入了"深度内容"

### 1.3 核心决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| **网站框架** | Astro 静态站点生成器 | 输出纯静态 HTML/CSS/JS，无框架痕迹；支持模板化开发；完美适配分层隐藏内容 |
| **样式方案** | 纯 CSS（手写，贴近真实小众站点风格） | 不用 Tailwind 等现代框架，避免"太像大厂产品"的违和感 |
| **客户端逻辑** | 原生 JavaScript（少量 TypeScript 辅助开发） | 谜题验证、关键词触发、进度存储全部用原生 JS，无框架依赖 |
| **状态管理** | localStorage + 简单的 JS 状态对象 | 探索进度、谜题完成状态、分支标记、结局触发 |
| **部署** | Vercel / Netlify 静态托管 | 免费、免备案、自动 CI/CD |
| **后端** | 无（纯静态站点） | 所有谜题为确定性客户端逻辑，无需服务端 |
| **仓库结构** | 单仓库（非 monorepo） | 只有一个网站项目，结构简单清晰 |

### 2.4 关键技术实现

#### 2.4.1 搜索关键词触发（search.ts）

```typescript
// 搜索框监听特定关键词，触发隐藏页面跳转
const TRIGGER_KEYWORDS: Record<string, string> = {
  '深海裂隙': '/trigger/rift',
  '星门坐标': '/trigger/stargate',
  '布景基地': '/trigger/base',
  // 更多触发词...
};

// 搜索时检测，命中则跳转到隐藏页面
function handleSearch(query: string): void {
  for (const [keyword, url] of Object.entries(TRIGGER_KEYWORDS)) {
    if (query.includes(keyword)) {
      // 记录发现
      recordDiscovery(url);
      // 跳转
      window.location.href = url;
      return;
    }
  }
  // 正常搜索：返回"无结果"或引导到公开页面
  showNoResults(query);
}
```

#### 2.4.2 谜题验证（puzzle.ts）

```typescript
// 所有谜题答案存储 SHA-256 哈希，不存明文
interface Puzzle {
  id: string;
  answerHash: string;        // SHA-256 hash
  clueLocations: string[];   // 线索所在的页面 URL
  unlockEffect: () => void;  // 解锁后的效果
  stage: number;
}

async function verifyAnswer(puzzleId: string, attempt: string): Promise<boolean> {
  const puzzle = PUZZLES.find(p => p.id === puzzleId);
  if (!puzzle) return false;
  const attemptHash = await sha256(attempt);
  return attemptHash === puzzle.answerHash;
}
```

#### 2.4.3 探索进度追踪（progress.ts）

```typescript
// 加权计算探索百分比
function calculateProgress(progress: GameProgress): number {
  const weights = {
    pagesVisited: 0.25,      // 公开页面浏览
    hiddenPagesFound: 0.30,  // 隐藏页面发现
    puzzlesSolved: 0.30,     // 谜题破解
    filesAccessed: 0.15,     // 文件下载/查看
  };

  const pagesScore = progress.discoveredPages.length / TOTAL_DISCOVERABLE_PAGES;
  const hiddenScore = countHiddenPages(progress.discoveredPages) / TOTAL_HIDDEN_PAGES;
  const puzzleScore = countSolved(progress.solvedPuzzles) / TOTAL_PUZZLES;
  const filesScore = progress.accessedFiles.length / TOTAL_FILES;

  return Math.round(
    (pagesScore * weights.pagesVisited +
     hiddenScore * weights.hiddenPagesFound +
     puzzleScore * weights.puzzlesSolved +
     filesScore * weights.filesAccessed) * 100
  );
}
```

#### 2.4.4 结局判定引擎（ending.ts）

```typescript
function determineEnding(progress: GameProgress): EndingType {
  const { explorationProgress, hasFullEvidence, hasAlertedPolice,
          hasRescuedRoommate, playerTargeted } = progress;

  // 结局四：完美结局（全部条件满足）
  if (explorationProgress >= 100 && hasFullEvidence &&
      hasAlertedPolice && hasRescuedRoommate) {
    return 'perfect_victory';
  }

  // 结局三：局部救赎
  if (explorationProgress >= 70 && hasAlertedPolice &&
      hasRescuedRoommate && !hasFullEvidence) {
    return 'partial_victory';
  }

  // 结局二：孤影救赎
  if (hasRescuedRoommate && !hasAlertedPolice) {
    return 'lone_rescue';
  }

  // 结局一：沉沦囚笼（默认 / 玩家被标记且探索不足）
  if (playerTargeted && explorationProgress < 50) {
    return 'captured';
  }

  // 默认走向结局一
  return 'captured';
}
```

## 三、开发模式

### 3.1 开发方法论

采用**增量式敏捷开发**，按游戏五大阶段拆分迭代：

- **Sprint 周期**：1-2 周/迭代
- **每个 Sprint 产出**：一个可独立游玩和测试的游戏阶段
- **MVP 原则**：Phase 0（站点框架 + 第一阶段内容）为最小可行产品

### 3.2 Git 工作流

```
main
  ├── feat/phase0-site-foundation    # 阶段0：站点框架
  ├── feat/phase1-stage1             # 阶段1：第一章内容
  ├── feat/phase2-stage2             # 阶段2：第二章内容
  ├── feat/phase3-stage3             # 阶段3：第三章内容
  ├── feat/phase4-endings            # 阶段4：结局系统
  └── feat/phase5-polish             # 阶段5：打磨部署
```

- **主分支**：`main`（保护分支，仅通过 PR 合并）
- **功能分支**：`feat/<feature-name>`
- **提交规范**：Conventional Commits（`feat:`, `fix:`, `docs:`, `refactor:`, `chore:`）

### 3.3 代码质量保障

| 层面 | 工具 | 用途 |
|------|------|------|
| 类型检查 | TypeScript | 谜题逻辑、数据定义的类型安全 |
| 代码规范 | ESLint + Prettier | 统一代码风格 |
| 单元测试 | Vitest | 谜题校验、结局判定、进度计算 |
| E2E 测试 | Playwright | 完整游玩流程（4 种结局路径） |
| 构建检查 | Astro build | 确保所有页面正常生成 |

### 3.4 开发环境

```bash

Node.js >= 18
pnpm >= 8

# 启动开发
pnpm install
pnpm dev              # 启动 Astro 开发服务器 (localhost:4321)

用途：按照游戏谜题系统的约定，创建一个新的谜题
输入：谜题类型（base64/caesar/keyword/stegano/coordinate）、所在阶段、难度等级
输出：谜题数据定义 + 线索文本 + 答案哈希 + 解锁逻辑
规范：
  - 所有谜题答案通过 SHA-256 哈希存储
  - 线索必须在叙事上下文中有呼应
  - 谜题难度逐阶段递进
  - 每条谜题至少提供 2 条不同维度的线索
```

### 5.2 Skill: `content` — 撰写游戏叙事

```
用途：撰写符合游戏世界观和角色设定的网站内容
输入：内容类型（社区帖子/日志/点位描述/后台记录）、所属阶段、视角角色
输出：符合人设语气的中文文本
规范：
  - 保持"林屿"和"沈辞"人设一致性
  - 所有看似超自然的现象必须有人工解释
  - 线索文本暗示而不明示
  - 匹配小众爱好者社区的语言风格
```

### 5.3 Skill: `test-flow` — 验证游戏流程

```
用途：测试指定阶段的游戏逻辑流程是否正确
输入：目标阶段、结局路径
输出：流程步骤清单、状态变更日志、潜在逻辑漏洞报告
```

### 5.4 Skill: `review-puzzle` — 谜题审查

```
用途：审查谜题的可解性和难度合理性
输入：谜题 ID
输出：可解性评估、难度评级、改进建议
```

## 六、项目 Rules 约定（CLAUDE.md 核心规则）

### 6.1 代码架构规则

```markdown
## 目录职责
- src/pages/ — Astro 页面文件，每个文件对应一个路由
- src/components/ — 可复用 Astro 组件
- src/scripts/ — 客户端 JavaScript（谜题、进度、结局）
- src/data/ — 游戏内容数据（纯 TS 对象，不含 UI 逻辑）
- src/styles/ — CSS 样式文件
- src/assets/ — 静态资源（图片、字体、下载文件）
- public/ — 直接映射到网站根目录

## 页面命名
- 公开页面：描述性命名（alien/, ruins/, community/）
- 隐藏页面：放在 hidden/ 或 trigger/ 子目录下
- 结局页面：放在 ending/ 子目录下

## 状态管理
- 所有游戏进度存储在 localStorage，键名使用 `arg_progress`
- 谜题答案仅存储 SHA-256 哈希值
- 状态读写通过 src/scripts/progress.ts 统一接口
```

### 6.2 命名规范

```markdown
## 文件命名
- Astro 页面/组件：kebab-case（如 search-bar.astro）
- TypeScript 脚本：kebab-case（如 puzzle-verify.ts）
- 数据文件：kebab-case（如 shenci-journal.ts）
- CSS 文件：kebab-case（如 home.css）

## 变量命名
- 谜题 ID：snake_case（如 stage1_login_password）
- 页面路径：kebab-case（如 /alien/sighting-001）
- 函数命名：camelCase（如 verifyAnswer）
- localStorage 键名：snake_case（如 arg_progress）
```

### 6.3 内容创作规则

```markdown
## 语言
- 所有面向玩家的文本使用简体中文
- 代码注释：中文（方便策划阅读）
- Git 提交信息：中文

## 叙事一致性
- "林屿"视角：谨慎细腻的观察者（体现在网站文案的风格）
- "沈辞"视角：大胆执着的行动派（体现在其日志/投稿的语气）
- 所有"超自然现象"在后续层级的页面中揭示为人工骗局
- 线索碎片化：每个页面/帖子提供部分信息，拼合理解全貌

## 谜题线索规范
- 每个谜题至少 2 条不同维度的线索
- 线索来源多样化：页面正文、HTML 源码、图片隐写、控制台消息
- 引导方式：暗示（如"有些答案藏在表面之下"）而非明示（"查看网页源代码"）
```

### 6.4 测试要求

```markdown
## 必须测试
- 所有谜题校验逻辑（正确/错误/边界）
- 结局判定引擎（四种结局 + 边界条件）
- 探索百分比加权计算
- localStorage 状态持久化与重置

## 推荐测试
- 关键词触发跳转正确性
- 隐藏页面 URL 不可通过公开链接访问
- 多周目状态重置完整性
```

## 七、关键风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 网站看起来太"假" | 玩家无法代入 | 参考真实国内小众论坛（如牧夫天文论坛、8264户外）；手写 CSS 不用现代框架 |
| 谜题难度失衡 | 卡关或太过简单 | 每道谜题独立盲测；提供二级间接线索作为安全网 |
| 隐藏页面被搜索引擎收录 | ARG 体验被破坏 | robots.txt 屏蔽 hidden/trigger/ending 目录；页面加 noindex meta |
| Astro 构建痕迹暴露 | 技术感太强 | 检查构建产物，移除 Astro 默认注入的注释/属性 |
| 多结局逻辑混乱 | 错误触发结局 | 结局判定为独立纯函数，单元测试覆盖所有分支组合 |
