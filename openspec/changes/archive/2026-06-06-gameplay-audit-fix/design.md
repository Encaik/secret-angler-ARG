## Context

当前搜索结果页（`search.astro`）的功能已经完整——历史存档检索、隐藏页面缓存、用户搜索、页面导航都有。但从"模拟真实搜索引擎"的角度看，有几处细节偏离了真实搜索引擎的体验。这些不是游戏机制问题，而是站点拟真度问题。

## Goals / Non-Goals

**Goals:**

- 搜索结果计数使用搜索引擎常用格式（`找到约 N 条结果`）
- 相关搜索推荐始终显示（当有匹配时），不只在卡关时出现
- 相关搜索推荐词根据当前查询词动态匹配，而非硬编码
- 页面导航结果显示可读标题，而非仅 URL 路径
- 补充拼音/缩写等边缘搜索词的映射

**Non-Goals:**

- 不改变搜索权限机制（kd7f3g 逻辑完全不动）
- 不改变谜题答案、密码、URL 路由
- 不添加任何游戏化 UI
- 不改变结局判定逻辑

## Decisions

### 决策 1：相关搜索始终显示

**选择**：每次搜索时，根据查询词从 `SEARCH_SUGGESTIONS` 映射表匹配推荐词，始终显示在结果底部。卡关时（`?hint=1`）使用 `stuck` 级别的推荐词替代 `normal` 级别。

**理由**：真实搜索引擎（百度/Google）对几乎所有搜索都显示相关搜索。当前仅在卡关时显示，且内容硬编码——这让"相关搜索"读起来像"游戏提示"而非搜索引擎功能。始终显示且动态匹配让该功能融入背景。

**实现**：

- 移除 `if (showHint)` 条件，改为始终渲染相关搜索区域（当有匹配时）
- 根据 `?hint=1` 参数选择推荐词级别：`hint=1` 时用 `stuck` 推荐，否则用 `normal` 推荐
- 推荐区域样式保持不变（灰色链接列表，与当前一致）

### 决策 2：搜索结果计数格式简化

**选择**：将当前分段计数格式（`N条历史存档、M条缓存页面...`）替换为搜索引擎风格的总数格式：`找到约 N 条结果`。

**理由**：当前格式暴露了内部数据结构（"历史存档"、"缓存页面"、"相关板块"等分类），让搜索结果页读起来像开发者面板。真实用户不需要知道结果是按什么分类存储的。

**实现**：

- 计算所有结果类别的总数
- `statusEl.textContent = \`找到约 ${totalResults} 条结果\``
- 各分类标题保留（`📜 历史存档`、`🌐 页面缓存` 等），帮助用户浏览

### 决策 3：页面导航结果显示标题

**选择**：在 `search-data.ts` 中新增 `PAGE_TITLES` 映射（URL → 页面标题），搜索结果渲染时显示标题而非裸 URL。

**理由**：当前页面导航结果仅显示 `/alien/alien-001/` 这样的路径。真实搜索引擎会显示页面标题 + URL。显示标题让结果更可读，同时不暴露内部路径结构。

**实现**：

- `search-data.ts` 中新增 `PAGE_TITLES: Record<string, string>`，为所有 `MOCK_RESULTS` 中出现的 URL 提供中文标题
- `search.astro` 渲染时读取标题，格式为：标题（链接）+ URL（灰色小字下方）

### 决策 4：补充拼音和缩写搜索词

**选择**：在 `ARCHIVED_CONTENT` 和 `MOCK_RESULTS` 中为常见拼音/缩写增加关键字映射。

**理由**：部分玩家可能使用拼音搜索（如 `helanshan`），当前这些搜索返回空结果。一个真实的社区搜索引擎不会对合理的拼音查询完全无响应。补充这些映射确保合理的搜索尝试都得到反馈。

**实现**：

- `ARCHIVED_CONTENT` 中增加 `helanshan` → 指向贺兰山相关内容的映射
- `ARCHIVED_CONTENT` 中增加 `shenci` 相关搜索词的映射复核（已有部分，检查完整性）
- 不新增内容，仅增加关键词映射（多个关键词指向同一批内容）
